What it is

tamux is not just a terminal multiplexer with an LLM bolted on. It is a shared execution environment where the daemon owns terminal sessions, agent threads, task queue state, goal-run state, approvals, memory, and telemetry. Electron, the TUI, the CLI, MCP clients, and chat gateways all reconnect to the same durable runtime.

Mental model: think of tamux as an autonomous operating environment with terminal surfaces, not as a disposable prompt window.

Capabilities

Daemon as source of truth

The daemon owns sessions, threads, tasks, approvals, goal runs, transcripts, memory, and operational state. Clients are views into that state, not the owners of it.

Durable goal runners

Goal runs accept long-running objectives, create plans, dispatch child tasks, survive disconnects, and record reflections, memory updates, and reusable skills.

PTY and workspace control

tamux manages panes, surfaces, terminals, command history, transcripts, and terminal-native execution while keeping agent actions visible and recoverable.

Layered memory

Identity, durable facts, operator profile, structured runtime state, recall, skills, and provenance-backed memory are all part of the system’s persistence model.

Governed autonomy

Risk is evaluated structurally, approvals are scope-bound and can go stale, and risky actions can be critiqued and constrained before execution.

Runtime plugin system

Plugins can add components, commands, YAML views, assistant tools, and executors, turning tamux into a host for domain-specific vertical workflows.

Multiple operator surfaces

Use Electron for the broadest desktop surface currently available, the TUI for SSH-friendly control, the CLI for automation, MCP for external agents, and gateways for chat platforms. Surface depth varies, but they reconnect to the same daemon state.

Self-orchestrating agent runtime

The daemon agent can delegate, checkpoint, detect stuck states, recover, learn from traces, and escalate when work gets blocked or risky.

Reality check: the daemon/runtime is the most complete layer today. Public clients already expose meaningful goal, approval, and inspection workflows, but some richer desktop Mission Control-style shells are still evolving on top of the shipped backend.

         |
         v
    tamux daemon

         +--> PTY session management
         +--> Agent runtime
         +--> Task queue
         +--> Goal runners
         +--> Approval flow
         +--> Persistence + telemetry

This is the fundamental boundary to understand. Once the daemon is running, the rest of the platform becomes interchangeable surfaces over the same long-lived state.

Who it's for

  • Developers who want long-running AI work to survive closing the UI.
  • Operators who need a visible, governed control plane rather than invisible automation.
  • Teams exploring multi-agent workflows, plugins, background tasks, and domain-specific mission layers.
  • People who want terminal-native workspaces but do not want to give up modern orchestration and memory.

Hosting

This docs site is plain HTML, CSS, and JavaScript. There is no mandatory docs generator and no build step. You can open index.html locally, serve the folder with a static HTTP server, or publish it through GitHub Pages.

Recommended path: keep the site in github-docs/ during authoring, then publish that folder through a Pages branch or copy it into a Pages-served directory such as /docs when you are ready to ship.
