# Milestone v2.0: Plugin Ecosystem

**Status:** SHIPPED 2026-03-25
**Phases:** 14-20
**Total Plans:** 17

## Overview

Community-friendly declarative plugin system — anyone can extend tamux with JSON manifests that declare commands, settings, API contracts, and bundled skills. Plugins install via CLI/npm and work across TUI and Electron. Validated end-to-end with a real Gmail/Calendar integration using Google OAuth2.

## Milestone Summary

**Key Decisions:**
- Declarative JSON manifests only — no executable plugin code (security)
- Hybrid storage: manifest on disk, metadata in SQLite
- Two separate plugins (gmail + calendar) in one npm package
- Handlebars template engine for API request/response transformation
- AES-256-GCM encryption for OAuth tokens at rest
- ON CONFLICT DO UPDATE instead of INSERT OR REPLACE (prevents CASCADE deletion)

**Issues Resolved:**
- INSERT OR REPLACE triggered ON DELETE CASCADE, wiping plugin settings on daemon restart
- TUI base modal handler intercepted Enter key for plugin settings fields
- xdg-open browser stderr corrupted TUI alternate screen
- Concierge LLM call blocked single-connection handler
- Skills were .yaml instead of .md (agentskills.io format)
- JSON body templates broke when Handlebars expressions contained quotes

**Issues Deferred:**
- Concierge welcome should be async (background task + broadcast)
- Test Connection should check OAuth credential status for auth plugins
- Plugin marketplace / hosted registry (v3 scope)
- Executable plugin code / WASM sandboxing (v3 scope)

**Technical Debt:**
- Body templates cannot use `default` helper with empty string fallback (JSON escaping issue)
- TUI plugin list not pushed on plugin install (needs daemon broadcast)
- Concierge generates new welcome on every connect (should cache when nothing changed)
